02 / 09

功能主线

核心不是新增一个配置项，而是让 analyze 到 compare 的每个阶段共享同一套语义结论和执行证据。

01 · ANALYZE

读取 changed assets、lineage、DDL、task 和 model，构造表级资产快照。

02 · RESOLVE

按用户声明、上游传播、严格自动规则、默认 unknown 的优先级判定。

03 · ROUTE

equivalent 验证本表；changed / unknown 沿下游寻找权威或观察性边界。

04 · SHADOW

用已验证 bundle root 重算最小作业集合，并发布 execution marker。

05 · COMPARE

验证 plan、workspace、shadow 和 QA marker，再解释 count / row compare。

轻量 replan：用户看到 warning 后可用 semantic-mode set --run … 补充选择。只有 workspace 和 analysis inputs 都未变化时才复用已有分析；否则要求重新 analyze。

04 / 09

五个关键安全边界

第二轮 Review 的主要价值在这里：不仅检查“能否跑通”，还检查等价证明、持久化证据和跨 worktree 执行是否可能说谎。

A · RENAME PROOF

只归一化可证明的目标

完整 relation identity + mutation target 检查，阻止把 ext.same_name 外部源表连带改名后误判 equivalent。

B · PLAN FRESHNESS

核心 API 自身 fail closed

run_shadow_plan() 和 compare_shadow_results() 都执行 bundle freshness，不能靠绕过 CLI 规避。

C · WORKTREE BINDING

验证 A 就必须执行 A

FreshPlanBundle 绑定 manifest root 与同一内存 plan；fingerprint v2 还摘要实际加载的工具源码。

D · EXECUTION PROVENANCE

本地结果不等于 QA 现状

UUID execution、项目锁和 QA marker 共同证明共享 QA 数据库仍属于该次 shadow-run。

E · COMPLETE COMPARE

抽样不能冒充权威通过

缺少 required check、只跑 count 或启用 sample 时，结果是 inconclusive，不是 passed。

持久化产物链路

用户声明不放进无限增长的项目级 decisions 文件，而是随 run 的 manifest 保存；plan 专门保存分析快照和解析后的语义事实。

manifest.json

run 上下文 + 用户 semantic intent；artifact 路径必须留在 run 内。

analysis inputs

baseline/current lineage 与 change analysis，各自有内容摘要。

plan.json

resolved semantics、jobs、anchors、checks、warnings、analysis snapshot。

shadow result

running → completed/failed，绑定 execution、workspace 和 plan。

compare result

验证状态、逐项结果、完整性契约与 shadow result fingerprint。

原子发布DDL 先内容寻址写入，plan 最后发布；成功后才清理旧 DDL。
主动失效新 plan 或 semantic intent 生成前删除旧 shadow / compare，失败不回退。
历史复用只有 table identity 与 semantic context fingerprint 同时匹配才复用声明。
07 / 09

Review 发现如何被关闭

下列是最值得重新检查的发现。完整清单见持久化 Review 报告；这里保留会影响正确性的主问题。

错误的 rename 等价证明

Critical

全局 identifier 替换可能把外部同名 source 一起改写，从而掩盖真实逻辑变化。

关闭方式：完整 relation identity、mutation target 限定、显式 model reference 字段和回归用例。

QA 执行来源不可信

Critical

completed JSON 不能证明共享 QA 数据库尚未被另一 run 重建。

关闭方式：UUID execution + 跨 worktree 本机锁 + QA marker exact match。

验证 A、执行 B

Critical

跨 worktree 直调可能校验 manifest root A，却从进程全局 root B 读取 SQL 或 model。

关闭方式：FreshPlanBundle root、fingerprint v2 runtime 源码摘要、rooted planner config。

Freshness 后二次加载

Important

freshness 和执行之间再次读取 plan，会产生并发 analyze 的 TOCTOU 窗口。

关闭方式：物化并消费同一份已验证内存 plan snapshot，DDL bytes 仍独立校验。

不完整 compare 误报通过

Important

只跑 count 或 sample row compare 时，匹配结果曾可能被解释为权威 passed。

关闭方式：required/executed checks + complete 标志；不完整统一 inconclusive。

最终复审

Closed

最终只读复审覆盖 rename、root、runtime、snapshot、manifest/DDL 路径与摘要约束。

结论：没有剩余 Critical、Important 或 Minor finding。

09 / 09

仍需知道的运行边界

这些不是本次未修复 finding，而是当前工具明确不承担的系统边界。Review 合入时应确认团队接受这些约束。

多主机并发

文件锁覆盖同一主机的不同进程和 worktree，不是分布式锁；多主机写同一 QA 仍需调度层保证单写。

生产数据快照

工具绑定执行代码和 plan，不冻结生产数据；shadow 与 compare 之间的生产数据变化仍可能产生真实差异。

Run 历史增长

声明随 run 保存，避免单一 decisions 文件无限追加；run 目录仍需按审计策略整体归档或清理。
