# Refact Run Session Design

## Goal

Provide a single refactor-session workflow that can freeze a pre-refactor
baseline, incrementally rebuild current lineage after each edit, score only the
changed and affected scope, report which issues were fixed or introduced, and
run data validation through an isolated QA database.

The workflow replaces the old idea of one large `refact_metadata.json` with
separate artifacts whose lifecycles are explicit.

## Command Model

Use one entrypoint:

```bash
python refact/run.py start --project shop
python refact/run.py analyze --manifest refact/runs/<run_id>/manifest.json
python refact/run.py analyze --manifest refact/runs/<run_id>/manifest.json --partition 2025-01-15
python refact/run.py shadow-run --manifest refact/runs/<run_id>/manifest.json
python refact/run.py compare --manifest refact/runs/<run_id>/manifest.json
```

Responsibilities:

- `start`: create a refactor run and freeze the baseline facts before edits.
- `analyze`: rebuild current facts, analyze changed scope, run scoped assessment,
  diff baseline/current issues, and refresh the validation plan.
- `analyze --partition`: manually constrain shadow-run and compare checks to a
  known partition value. There is no automatic Doris partition discovery in the
  default workflow.
- `shadow-run`: execute the validation plan in bypass mode, reading production
  inputs and writing affected outputs to the QA database.
- `compare`: compare production baseline results with QA shadow-run results.

## Run Directory

Each run owns all generated artifacts under:

```text
refact/runs/<run_id>/
  manifest.json

  baseline/
    lineage_data.json
    task_lineage_cache.json
    assess_result.json

  current/
    lineage_data.json
    task_lineage_cache.json
    assess_result.json

  analysis/
    change_analysis.json
    issue_diff.json

  verification/
    plan.json
    shadow_run_result.json
    compare_result.json
```

`baseline/*` is frozen after `start`. `current/*`, `analysis/*`, and
`verification/plan.json` are regenerated by each `analyze`.

## Manifest

`manifest.json` is the only stable input to later commands. It is an index and
baseline contract, not a container for large facts.

```json
{
  "run_id": "20260620_153000_shop",
  "project": "shop",
  "created_at": "2026-06-20T15:30:00+08:00",
  "base_git": {
    "branch": "codex/example",
    "head": "abc1234",
    "dirty": false
  },
  "artifacts": {
    "baseline_lineage": "baseline/lineage_data.json",
    "baseline_assess": "baseline/assess_result.json",
    "current_lineage": "current/lineage_data.json",
    "current_assess": "current/assess_result.json",
    "change_analysis": "analysis/change_analysis.json",
    "issue_diff": "analysis/issue_diff.json",
    "verification_plan": "verification/plan.json"
  }
}
```

If `start` is run from a dirty worktree, the manifest must record
`dirty: true`; reports should make clear that the baseline is not a strict
before-edit snapshot.

## Lineage

Lineage is not assumed to be tracked in git. `start` writes a full baseline
lineage file and a task-level lineage cache into the run directory.

`analyze` builds current lineage incrementally:

1. Parse all current DDL files and build the current schema.
2. For each task, compute a cache key from:
   - task source path,
   - task SQL content hash,
   - referenced schema slice hash,
   - lineage extractor version hash,
   - project catalog/database/config hash.
3. Reuse cached task lineage only when the key matches.
4. Recompute invalidated tasks.
5. Rebuild the full current lineage output from task lineage entries and the
   current schema.

The final current lineage must be rebuilt, not patched in place, so stale
tables, fields, and transient artifacts cannot survive from an older run.

## Scoped Assessment

Assessment is change-scoped, not limited to directly edited files.

The scope contains:

- directly changed DDL tables, task jobs, model tables, and config files;
- upstream/downstream graph neighbors needed to explain changed lineage;
- anchor tables and writer peers affected by changed outputs;
- global or semi-global dimensions when naming config, business semantics, or
  rule configuration changes.

`analyze` writes this to `analysis/change_analysis.json`:

```json
{
  "changed_files": [],
  "changed_assets": {
    "ddl_tables": [],
    "task_jobs": [],
    "model_tables": [],
    "config_files": []
  },
  "affected_scope": {
    "direct_tables": [],
    "downstream_tables": [],
    "anchor_tables": [],
    "assessment_tables": [],
    "assessment_tasks": [],
    "global_dimensions": []
  },
  "lineage_diff": {
    "added_edges": [],
    "removed_edges": [],
    "changed_tables": []
  }
}
```

`change_analysis.json` is an output artifact. `analyze` must not trust a previous
copy as input; it recomputes the file from the manifest, baseline artifacts, and
current worktree.

## Comparable Assessment Results

`assess_result.json` should be directly comparable. Each issue keeps its
display id and also includes a stable fingerprint.

```json
{
  "id": "model_design.iss_003",
  "fingerprint": "model_design|MODEL_DWS_GRAIN_PRESENT|table|dws_product_sales_daily",
  "rule_id": "MODEL_DWS_GRAIN_PRESENT",
  "target": {
    "type": "table",
    "name": "dws_product_sales_daily"
  }
}
```

`id` is display-oriented and can depend on output order. `fingerprint` is
machine-oriented and must be stable across runs.

Fingerprint format:

```text
dimension | rule_id | target.type | target.name | optional discriminator
```

Rule-specific discriminators include column name, metric group/name,
source-target edge, or logical file path. Dynamic message text, LLM confidence,
and other volatile evidence must not participate in the fingerprint.

## Issue Diff

`analysis/issue_diff.json` compares baseline and current `assess_result.json`
by issue fingerprint.

```json
{
  "summary": {
    "baseline_issue_count": 8,
    "current_issue_count": 3,
    "fixed_count": 6,
    "remaining_count": 2,
    "new_count": 1
  },
  "fixed_issues": [],
  "remaining_issues": [],
  "new_issues": [],
  "scope_score": {}
}
```

This file is a report artifact, not a source of truth.

## Verification Plan

`verification/plan.json` replaces the old broad `refact_metadata.json` concept.
It contains only data-validation execution details:

```json
{
  "project": "shop",
  "project_db": "shop_dm",
  "qa_db": "shop_dm_qa",
  "baseline_ddl": {},
  "ddl_changes": [],
  "partition_info": {},
  "jobs_to_run": [],
  "checks": []
}
```

`shadow-run` reads this plan, recreates QA structures, applies DDL changes, and
runs affected jobs with SQL rewritten so production inputs remain readable while
affected outputs land in the QA database.

`compare` reads the same plan and compares configured production and QA targets
with methods such as `count` and `row_compare`.

## Module Boundaries

Implementation should keep CLIs thin:

```text
refact/run.py                 # subcommand entrypoint
refact/session.py             # manifest and run-directory lifecycle
refact/incremental_lineage.py # refactor run lineage artifact builder
refact/change_analysis.py     # changed assets, scope, lineage diff
refact/scoped_assess.py       # scoped assess execution
refact/issue_diff.py          # fingerprint-based issue diff
refact/verification_plan.py   # build verification/plan.json
refact/shadow_run.py          # QA bypass execution
refact/compare.py             # production vs QA comparison
lineage/task_cache.py         # reusable task cache key and cache helpers
```

Existing logic from `lineage.lineage_extractor` and
`assess.assess_middle_layer` should be reused behind these boundaries where
practical. Shadow execution and production-vs-QA comparison live directly in
`refact/shadow_run.py` and `refact/compare.py`; there should not be separate
`verify_run` or `verify_check` core modules. Refactor-specific baseline/current
artifact paths stay in `refact/incremental_lineage.py`, while task-level lineage
cache fingerprints and cache entry helpers live under `lineage/task_cache.py`.
