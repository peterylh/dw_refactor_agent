# 数据集市建设质量评估 (assess-datamart)

评估数仓数据集市 DWD/DWS 中间层的建设质量，从复用度、链路长度、架构合理性、命名规范四个维度打分，输出评分明细与违规清单。
详细分析问题和改进建议供人工（或 agent）后续处理时参考。

## 使用场景

- 数仓中间层质量审计
- 分层架构健康度巡检
- 命名规范合规检查
- 发现分层错配（表名与真实层级不一致）

## 执行流程

1. 检测 llm 配置
echo "${DEEPSEEK_API_KEY:+DEEPSEEK_API_KEY 已设置}"

2. 执行assess_middle_layer检测脚本

```bash
python assess/assess_middle_layer.py --project {shop|olist} [--llm] [--no-cache]
```

- 结果写入 `assess/assess_result_{project}.json`
- `--llm` 参数可启用 LLM 智能分层巡检（如果运行环境用 `DEEPSEEK_API_KEY` 环境变量，则加上--llm）。LLM 会对每张 DWD/DWS 表推断真实层级和表类型(dimension/fact)，检测出的更多违规（分层错配、维度表位置不当）
- `--no-cache` 可强制重新调用LLM API。 LLM 调用代价大，尽量不要使用no-cache,除非你知道 cache 不是最新的

3.  展示总结报告
报告格式要求：
- 展示整体评分
- 分四个维度详细展示每个维度的评分标准和得分
- 按优先级给出解决建议

## 四个评分维度

### 1. 复用度 (权重 25%)

评估中间层表被下游引用的次数。

| 引用次数 | 得分 | 含义 |
|---------|------|------|
| ≥3 | 100 | 高复用，公共中间层价值充分 |
| 1-2 | 33.3 | 低复用，仅为单一下游服务 |
| 0 | 0 | 无引用，可能是死表或应归入 ADS |

**分析要点**：
- 0 引用的表：是死表，还是应该归入 ADS 层？（如下游=0 且含聚合→大概率是 ADS）
- 1-2 引用的表：能否合并相似场景提升复用度？
- 分析 distribution 分布（高复用/一般/无引用）

### 2. 链路长度 (权重 25%)

评估 ADS 到 ODS 之间经过的中间层深度。

| 深度 | 得分 | 含义 |
|------|------|------|
| 2 | 100 | DWD+DWS 两层完整 |
| 1 | 50 | 缺失 DWD 或 DWS |
| 0 | 0 | 无中间层，ODS 直达 ADS |
| ≥3 | 30 | 链路过长，存在冗余 |

**分析要点**：
- depth=1 的 ADS 表：跳过了哪一层？是否有意为之？
- depth≥3 的 ADS 表：是否存在可精简的冗余中间层？

### 3. 架构合理性 (权重 25%)

综合检测表间依赖违规和 LLM 分层推断违规，扣分项共三类：

**血缘依赖违规**（基于 DAG 拓扑检测）：

| 违规类型 | 严重度 | 单次扣分 |
|---------|--------|---------|
| 反向依赖(ADS→ODS) | 严重 | 40 |
| 反向依赖跳过两层 | 严重 | 30 |
| 反向依赖跳过一层 | 高 | 20 |
| 同层依赖 | 低 | 2 |
| 跳过中间层(DWD→ADS) | 低 | 5 |
| 跳过两层(ODS→ADS) | 中 | 10 |

**LLM 推断违规**（需启用 `--llm`，结果仅供参考）：

| 违规类型 | 严重度 | 单次扣分 |
|---------|--------|---------|
| 维度表位置不当：LLM 判定为 dimension 类型但表名非 `dim_*` | 低 | 5 |
| 分层错配：表名前缀与 LLM 推断层级不一致 | 中 | 10 |

> 原始分经分段线性映射转换为展示分，高分段差异更敏感。

**分析要点**：
- 跳过中间层违规通常是复用度低的直接原因：ADS 绕过 DWS 直接读 DWD，导致 DWS 失去复用场景
- 维度表位置不当：LLM 主要依据 DDL 字段特征（无度量字段、主键为实体 ID）和 ETL 逻辑（无聚合）判定，应结合业务确认是否将 `dwd_*` 迁移至 `dim_*`
- 分层错配：LLM 综合下游引用数、ETL 聚合特征、血缘深度判断，置信度 < 0.8 时建议人工复核

### 4. 命名规范 (权重 25%)

检查表名和字段名是否符合 `naming_config.yaml` 规范。

**表名检查**：前缀是否匹配所在分层的命名模板（`ods_` / `dwd_` / `dws_` / `ads_` / `dim_`）。注意：表名检查依据的是**表当前声明的层级**，与 LLM 推断无关。

**字段名检查**：是否匹配 `suffix_field` / `prefix_field` 或 `common_columns`。

**分析要点**：
- 列出得分最低的几张表及不合规字段清单
- "无匹配模式"字段占比：高占比说明 `naming_config.yaml` 的后缀列表需要扩充
- 维度属性字段（address, city, age, gender 等）天然难以用后缀规范覆盖，更适合加入 `common_columns` 白名单管理

## 改进建议分级

**P0 — 严重/高危**：
- 存在反向依赖 → 重构 ETL 链路
- LLM 检测到分层错配（`is_violating_current_name: true`，置信度 > 0.8）→ 重命名表或调整 ETL 目标层

**P1 — 中优先级**：
- 复用度为 0 的表 → 确认是否死表，或应归入 ADS
- depth=0 的 ADS 表 → 补建中间层
- 命名规范严重偏离的表（得分 < 50）→ 添加命名规则或重命名字段
- 维度表位置不当（LLM 判定为 dimension 的表）→ 评估是否迁移至 DIM 层

**P2 — 低优先级**：
- 复用度 1-2 的表 → 分析是否可以扩展下游场景
- 跳过中间层违规 → 将聚合逻辑下沉至 DWS 层
- 少量字段命名不合规 → 补充 `naming_config.yaml` 后缀或 `common_columns`
- LLM 置信度 < 0.8 的分类结果 → 人工复核
